Code d'honneur
En vous inscrivant à ce cours, vous vous engagez à respecter les règles suivantes :

Règle 1:	Vous devez être l’auteur·rice des solutions que vous soumettez.
Si vous copiez le travail d’une autre personne ou si vous vous en inspirez fortement en omettant de le mentionner, vous commettez un acte de plagiat. Par exemple, le fait d’utiliser une solution trouvée dans le web ou la solution d’un·e camarade de votre classe ou d’une année précédente constitue une violation de la présente règle. D’une manière générale, vous ne devez pas chercher à consulter une solution existante à un problème qui vous est posé dans le cadre de ce cours.
Règle 2:	Vous ne devez pas partager vos solutions avec vos camarades.
Vous ne devez en aucun cas partager vos solutions avec vos camarades, que ce soit à leur demande expresse ou en les publiant d’une manière ou d’une autre.
Règle 3:	Vous devez mentionner toute aide que vous recevez.
Si vous recevez de l’aide de la part d’une personne autre que votre enseignant·e pour la réalisation d’un travail, vous devez indiquer la nature de cette aide, et la manière dont elle a influencé votre solution. En outre, vous devez veiller à ce que l’aide reçue ne viole pas les règles 1 et 2.
Règle 4:	Vous devez être en mesure d’expliquer dans leur ensemble les solutions que vous soumettez.
Quelle que soit l’aide que vous recevez pour effectuer un travail, vous devez pouvoir fournir des explications sur l’ensemble de la solution. De plus, dans le cas d’un travail de groupe et sauf indication contraire, vous devez être en mesure d’expliquer non seulement vos propres contributions, mais également celles de vos partenaires.
Règle 5:	Vous devez effectuer vos travaux de manière consciencieuse.
Pour tous les travaux qui vous sont demandés, vous devez soumettre en temps et heure une solution réalisée avec minutie et application. Dans le cas où vous n’arriveriez pas au bout d’un travail, vous devez rendre la solution inachevée et un compte rendu détaillé des problèmes rencontrés.

Objectifs du module
Compétence
Analyser des stocks d’informations provenant de diverses sources, caractériser les données et les classer. À partir des indices obtenus lors de l'analyse, élaborer un modèle de données conceptuel et transformer celui-ci en un modèle de données logique relationnel.

Objectif 1
Examiner et classer les données issues de divers stocks d’informations structurés ou non structurés.
Connaissances opérationnelles nécessaires:
Connaître différentes formes de données (numériques: données de mesure/classifications; verbales: textes sous forme écrite ou orale; non verbales: musique, photos, vidéos, etc.) et leur mode de sauvegarde sur un support de données. Connaître les traitement dont elles peuvent l'objet dans un système informatique.
Connaître les caractéristiques structurelles de diverses sources d’informations (documents texte, feuilles de données, graphiques de présentation, bases de données, etc.) et les conséquences sur les possibilité de traitement électronique de ces sources de données.
Objectif 2
Caractérise un ensemble d'informations pour le traitement en termes de qualité, par exemple l'exhaustivité, l'unicité, l'incohérence, la redondance et nettoie les données si nécessaire.
Connaissances opérationnelles nécessaires:
Connaît les caractéristiques d'une base de données en termes d'unicité et d'exhaustivité (différentes indications pour une même information, différents contenus d'information).
Connaît les types de données répandus pour caractériser les données (nombre entier, nombre rationnel, heure, date, texte, booléen, etc.)
Connaît les différents types d'échelle (nominale, ordinale et métrique) et les conséquences qui en découlent pour les possibilités d'évaluation.
Objectif 3
Sélectionne les informations qui se prêtent à une évaluation et tient compte des aspects relatifs aux éventuels besoins de protection (p. ex. données personnelles, confidentialité).
Connaissances opérationnelles nécessaires:
Connaît les types de diagrammes courants pour la représentation de données dans des graphiques commerciaux (diagramme circulaire, diagramme à barres, diagramme linéaire, diagramme en réseau, etc.)
Connaît la nécessité d'évaluer la pertinence des résultats d'évaluation et le fait que les tendances de ces derniers peuvent être pertinentes ou non, selon la perspective.
Connaît les caractéristiques statistiques importantes (minimum, maximum, moyenne, médiane, moyenne, nombre d'observations) et les conditions (types d'échelle, nombre d'observations, qualité des données, etc.) pour une déclaration fiable.
Connaît les principes de base du secret et de la protection des données lors de la saisie, de l'évaluation et de la présentation de bases de données à protéger.
Objectif 4
Crée un modèle de données conceptuel pour les données nettoyées et définit les types d'entités, les attributs et les relations.
Connaissances opérationnelles nécessaires:
Connaît la procédure méthodique pour le développement d'un modèle conceptuel de données (Entity Relationship Model, ERM), en partant des structures de données ou des exigences du client (spécialiser, généraliser, identifier et définir les types d'entités, définir les attributs, décrire les relations).
Connaît les éléments d'un modèle conceptuel de données (type d'entité, attribut, association, cardinalité), leurs propriétés et leur représentation.
Connaît les différentes cardinalités des associations (1:1, 1:m, n:m) et connaît la procédure permettant de déterminer le type de relation entre les structures de données.
Objectif 5
Transforme le modèle de données conceptuel en un modèle de données logique et relationnel en complétant les clés d'identification et étrangères, les types de données et les éventuelles tables intermédiaires.
Connaissances requises
Connaît les différences entre le modèle de données conceptuel et le modèle de données logique et relationnel (Relational Model, RM) et la procédure pour passer du modèle de données conceptuel au modèle de données logique et relationnel.
Connaît les caractéristiques à remplir par les clés d'identification et les possibilités de traitement qui en découlent en tant que clés primaires et étrangères.
Connaît la nécessité des tables intermédiaires pour la représentation des cardinalités n:m entre les types d'entités.
Connaît les éléments de construction étendus (p. ex. association récursive) du modèle de données logique et relationnel.
Objectif 6
Normalise le modèle de données logique et relationnel.
Connaissances requises
Connaît les étapes de la normalisation de la forme normale 0 à la forme normale 3.
Connaît les raisons pour lesquelles une normalisation complète jusqu'à la 3e forme normale n'est pas toujours recherchée au profit d'une meilleure performance.
Objectif 7
Reproduit le modèle de données relationnel conceptuel et logique dans une représentation appropriée.
Connaissances requises
Connaît les notations pour la représentation de modèles de données relationnels conceptuels et logiques (notation IEM ou Crow foot, Chen, Modified Chen, Zehnder, UML, etc.)
Connaît des outils permettant de créer des modèles de données relationnels conceptuels et logiques.
Références bibliographiques
Andreas Meier (auth.), Introduction pratique aux bases de données relationnelles. Springer Paris, 2006.
Modalités d'évaluation
L'évaluation prend la forme de trois tests de connaissance d'environ 45 min. De petits travaux personnels peuvent également être demandés et notés.

Durant les tests, les apprentis ont droit à une refcard personnelle qu'ils font évoluer tout au long du module. Le contenu de la refcard doit être réalisé de manière individuelle et doit tenir sur une page A4 (un côté d'une feuille A4).
